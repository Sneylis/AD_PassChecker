# ARCHITECTURE — устройство системы и обоснование решений

## 1. Почему офлайн-аудит через IFM, а не онлайн?

DSInternals `Test-PasswordQuality` читает **NTDS.dit** напрямую. В
онлайн-режиме это означает доступ к живой базе AD с правами, близкими
к Domain Admin. Регламент требует работу под **Backup Operator**:
эта роль достаточна, чтобы запустить `ntdsutil ... ifm` (снимок в
согласованном состоянии + SYSTEM hive), но НЕ даёт прав на изменение
AD.

Таким образом IFM — это:
* снимок-копия, работа над которой не влияет на прод AD;
* ограниченные права (Backup Operator, а не DA);
* чистое разделение "кто делает снимок" (DC) и "кто его анализирует"
  (изолированная станция);
* возможность унести снимок в **изолированный VLAN** и обработать
  там, не давая станции сетевого доступа к DC.

## 2. Почему AES-256 7z, а не BitLocker-контейнер или PGP?

* 7z с `-mhe=on` шифрует **и** содержимое, **и** имена файлов.
* Бинарь 7z — проверенный, повсеместно распространённый, не тянет
  .NET-зависимости.
* BitLocker-контейнер требует административных прав на DC для
  создания, плюс завязка на TPM / recovery key.
* PGP красив, но добавляет зоопарк ключей и сложность
  доставки public key на DC.

Ключ — один **симметричный AES-256 base64**, генерируется один раз
на станции, передаётся на DC через **CMS-pkcs7** (асимметрично,
чтобы не светить симметрику по проводам).

## 3. Почему CMS для передачи ключа, а не Windows DPAPI?

Windows **DPAPI per-user** привязан к профилю конкретного пользователя,
а нам надо "DC должен расшифровать под своей машинной учёткой".

**DPAPI per-machine** (через LSA) привязан к конкретной машине —
ключ, зашифрованный станцией под машинной учёткой, DC никогда не
расшифрует.

**CMS (PKCS#7)** — стандарт, работает по схеме:
1. Станция берёт **публичный сертификат** DC (любой machine cert,
   например LAPS cert или выделенный KerbAuth).
2. Зашифровывает AES-ключ под этим публичным ключом.
3. DC своим **private key** расшифровывает.

Это штатный `Protect-CmsMessage` / `Unprotect-CmsMessage` в PS 5.1,
ноль внешних зависимостей.

## 4. Почему gMSA и Backup Operators, а не Domain Admins?

Минимум прав:
* `Backup Operator` достаточно для `ntdsutil ... ifm` (это и был
  оригинальный use-case этой роли).
* `Backup Operator` НЕ может модифицировать AD, менять пароли,
  создавать юзеров.
* `gMSA` — нет пароля, который кто-то может запомнить/украсть;
  ротация автоматическая; можно ограничить набор хостов, которым
  можно вытянуть пароль (`PrincipalsAllowedToRetrieveManagedPassword`).

## 5. Зачем 5 фаз, а не один большой скрипт?

Каждая фаза:
* имеет **единственную** зону ответственности (Single Responsibility);
* может быть запущена **изолированно** для диагностики
  (`Test-HIBPIntegrity.ps1 -ConfigPath ...` — без всех остальных);
* имеет **exit code**, который оркестратор преобразует в единый
  исход;
* пишет свой `PhaseName` в лог через `Set-AuditPhase` → легко
  грепать логи по фазе.

Отдельная **Phase 5** (cleanup) нужна, чтобы гарантировать
"финальный вентильный проход" даже если Phase 3 или 4 упали
(это `finally` оркестратора).

## 6. Почему SDelete 3-passes, а не один overwrite или Remove-Item?

* `Remove-Item` — только снимает указатель в `$MFT`; блоки с
  хешами паролей физически остаются на NVMe/HDD.
* Одиночный overwrite недостаточен по DoD 5220.22-M (хоть на
  современных SSD это условно).
* SDelete с `-p 3` реализует 3-проходную схему: `0xFF`, `0x00`,
  random; после — `TRIM` для SSD.

Альтернатива — `cipher /w:C:\AuditWork`, но она очень медленная и
работает на уровне тома.

## 7. Почему SecretStore, а не SecureString в файле?

* `ConvertTo-SecureString` / `Export-Clixml` привязывает шифрование
  к **одному конкретному пользователю** (DPAPI per-user).
  Scheduled Task под gMSA → другой пользователь → нельзя
  расшифровать.
* SecretStore с `Authentication=None` использует DPAPI **per-machine**
  → при условии BitLocker FDE это безопасно: ключ DPAPI хранится
  в LSA, вытащить его можно только из оперативки запущенной
  машины. А оперативка на BitLocker-станции с выключенным
  pagefile/crashdump **не протекает на диск**.

## 8. Зачем S/MIME для письма с паролем архива?

Чтобы получатель (security engineer) мог:
1. Убедиться, что письмо пришло именно от аудит-системы,
   а не от фишера.
2. Убедиться, что никто по пути не подменил тело с паролем.

Шифрование по получателю (CMS-Enveloped) делать не стали:
внутренний SMTP-relay заведомо не слушает внешний трафик,
а ложная сложность "получатель вне домена" нам не нужна.

## 9. Почему 5 дней retention, а не 30 / 90?

Требование пользователя: **"удалять результат через 5 дней"**.
Обоснование:
* отчёт нужен для актуального принятия решения "что менять сейчас";
* хранить дольше = хранить PII (имена юзеров) без практической
  пользы;
* архивный отчёт при необходимости долгосрочного хранения
  пересохраняется security engineer'ом вручную в отдельную
  корпоративную систему с нужной retention-политикой.

## 10. Почему correlation ID, а не просто timestamp?

Один прогон = 5 фаз на 2+ машинах. Искать по timestamp'у
муторно (время DC и станции не идеально синхронны).
CID = 8-символьный hex, одинаковый для всех записей одного прогона,
прошит в имя архива, в имя отчёта, в имя attestation.

Один `grep CID=abcd1234 *.log` на обоих хостах → полная
хронология одного аудита.

## 11. Матрица угроз → митигация

| Угроза | Митигация |
|---|---|
| Утечка NTDS.dit | 7z AES-256 + `-mhe=on`, шифрование в покое (BitLocker) |
| Подмена HIBP | SHA-256 verify перед каждым использованием |
| Компрометация станции | BitLocker, pagefile/crash dump off, изолированный VLAN, только исходящий SMB+SMTP |
| Компрометация DC | Это уже game-over независимо от аудита |
| Кража отчёта с SMB | 7z архив с уникальным паролем; пароль доставляется отдельно (S/MIME email) |
| Утечка пароля архива | Пароль одноразовый (каждый прогон свой), живёт только в SecureString + email; retention 5 дней |
| Вмешательство в scheduled task | Task запускается от gMSA с ограниченным списком хостов; изменение task'а пишется в 4702 Event Log |
| Долгоживущий ключ AES | Ротация раз в год (`Install-AuditStation -RotateKey`) |
| Ошибка (bug) аудита → оставленный .dit | Phase 5 сканирует residue и пишет signed attestation |

## 12. Чего НЕ делает эта система (и почему)

* **Не ломает пароли** brute-force'ом. Только сверяет с HIBP-словарём.
  Brute force — отдельная активность, требует согласований и
  в 99% случаев не нужна (HIBP покрывает 500M+ хешей).
* **Не рассылает уведомления пользователям** ("твой пароль слаб").
  Это зона ответственности IAM/HR, не аудита.
* **Не изменяет AD**. Только читает (IFM).
* **Не интегрирована с SIEM напрямую** (Splunk forwarder и т.п.).
  Но пишет критичное в Event Log `ADPasswordAudit` → любая агентная
  система (WEF, Sysmon, Splunk UF) подхватит сама.
